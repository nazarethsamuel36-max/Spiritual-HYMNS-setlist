Chapter 1: Introduction

1.1 Background
Music plays a vital role in congregational worship. Managing a library of hundreds of songs across different keys and languages is a significant logistical challenge. Paper songbooks are difficult to update, and static PDFs do not allow for flexible key changes during rehearsals or live performances. The Worship Song Library is a comprehensive digital management system designed to centralize, organize, and intelligently display worship songs across multiple languages, including Hindi, Marathi, and English.

1.2 Problem Statement
Traditional digital solutions for song management often fail in the following areas:
- Chord Drift: On different screen sizes, spaces used to position chords shift, causing chords to land over the wrong words.
- Language Barriers: Bilingual congregations need to see songs in both native script and Roman transliteration.
- Key Constraints: Transposing a song manually is error-prone and time-consuming for musicians.

1.3 Objectives
The primary objectives of this project are:
- To develop a relational database to store songs with structured metadata.
- To implement a chord parsing system that converts traditional formats into an anchored format.
- To create a responsive rendering engine that preserves chord alignment on mobile devices.
- To enable real-time transposition of song keys using mathematical logic.

1.4 Project Scope
The system focuses on the frontend display and backend management of song data. It includes features for song retrieval, search, setlist management, and live transposition. It is primarily a visual tool for musicians to use during live performances and rehearsals.

Chapter 3: Analysis (Existing vs Proposed)

3.1 Existing System Analysis
Three primary paradigms exist in the current market:
- Community Databases (Ultimate Guitar): Rely on space-based alignment which breaks on mobile screens (Chord Drift).
- Professional Apps (OnSong): Expensive, proprietary, and often restricted to specific hardware (iOS).
- Static Formats (PDF): Guarantee visual fidelity but offer zero flexibility for transposition or script toggling.

3.2 Proposed System (Worship Song Library)
The proposed system moves from a visual text paradigm to a structured data paradigm.
- Position-Based Mapping: Chords are anchored to character indices rather than spaces.
- Virtual Wrapping: A custom algorithm that splits lines while keeping chords in sync.
- Multi-Script Support: A single database entry supports both original and transliterated lyrics.

3.3 Comparative Justification
- Alignment: Space-based (Fragile) vs Position-based (Stable).
- Responsiveness: Native wrapping (Breaks sync) vs Virtual wrapping (Preserves sync).
- Transposition: Manual calculation (Slow) vs Algorithmic shifting (Instant).
- Data Model: Text blobs (Unstructured) vs Relational tables (Atomic).

Chapter 4: System Design

4.1 Database Design (ERD)
The system uses a highly normalized structure to ensure data integrity:
- Songs Table: Stores title, artist, language, and primary lyric strings.
- Sections Table: Divides songs into Verse, Chorus, and Bridge.
- Song Lines Table: Stores individual lines of lyrics for precise anchoring.
- Line Chords Table: Stores the chord symbol and the exact character index for placement.

4.2 Functional Modules
- Search Engine: Multi-dimensional search across title, number, and lyrics with weighted ranking.
- Transposition Engine: Mathematical utility that shifts chords along a 12-step chromatic scale.
- Rendering Engine: Client-side logic that interleaves chords and lyrics based on database indices.

4.3 User Interface Design
The UI is designed for high-pressure live environments:
- High Contrast: For readability on stage.
- Minimal Clicks: Search-to-Song transition is optimized for speed.
- Interactive Controls: Buttons for transposing (+/-) and script toggling are always accessible.

Chapter 5: Implementation

5.1 Chord Extraction Logic
The system implements a regex-based parser to ingest songs. When a song line like [G]Amazing [D]grace is imported, the system:
- Identifies the chord symbol inside brackets.
- Records the character position of the following word.
- Saves the chord and index as separate rows in the database.

5.2 Transposition Algorithm
Transposition is handled by a circular chromatic array:
- [C, C#, D, D#, E, F, F#, G, G#, A, A#, B]
- To transpose +2 steps, the system finds the current index, adds the offset, and uses modulo arithmetic to find the new chord symbol. This ensures 100% mathematical accuracy.

5.3 Virtual Wrapping Logic
To solve mobile responsiveness, the JavaScript engine:
- Measures the available container width.
- Identifies if a lyric line will overflow.
- Manually splits the line at a space character.
- Distributes the chords into the new sub-lines by recalculating their relative positions.

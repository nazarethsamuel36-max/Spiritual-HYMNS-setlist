4.15 Deep Dive: Songs Table

The songs table is the root entity and the anchor point for every other table in the schema. Understanding each field is essential for understanding the entire system.

4.15.1 Field: id (Primary Key)

Data Type: INT, AUTO_INCREMENT
Purpose: Provides a unique, immutable identifier for each song. This ID is used in every foreign key relationship and in every URL that references a song (e.g., /song?id=45).

Why AUTO_INCREMENT: Sequential integer IDs are the most efficient primary key type for MySQL's InnoDB storage engine. InnoDB stores data in a clustered index ordered by primary key. Sequential integers minimize page splits during INSERT operations, maintaining write performance as the library grows.

Why Not UUID: UUIDs (128-bit random identifiers) would provide globally unique IDs but would degrade InnoDB performance due to random insertion order. Since this application operates on a single database instance, sequential integers are sufficient and optimal.

4.15.2 Field: song_number

Data Type: INT, INDEXED, NULLABLE
Purpose: Stores the traditional songbook number that musicians use to identify songs. In Indian church songbooks, each song has a fixed number (e.g., Song #979 is always "Aarati Ho Aarati" regardless of which church or which edition of the book).

Why Separate from id: The song_number is a domain-specific identifier that may not be sequential, may have gaps (some numbers are retired), and may not be unique across languages (Hindi Song #1 and English Song #1 are different songs). The database id is a technical identifier; the song_number is a domain identifier. Conflating them would create design problems.

Why NULLABLE: Not all songs have traditional songbook numbers. New compositions or imported songs from online sources may not have an assigned number. Making this field nullable prevents forcing administrators to invent arbitrary numbers for unnumbered songs.

4.15.3 Field: lyrics_original vs lyrics_roman

These two fields represent the dual-script architecture of the system.

lyrics_original (TEXT, NULLABLE): Contains the lyrics in the song's native script. For Hindi and Marathi songs, this is Devanagari Unicode text. For English songs, this field is typically NULL because the Roman script IS the original script.

lyrics_roman (TEXT, NULLABLE): Contains the lyrics in Roman (Latin) transliteration. For Hindi and Marathi songs, this is the phonetic English rendering. For English songs, this contains the actual lyrics.

Design Decision: Both fields store the COMPLETE lyrics as a continuous text block. This is deliberately redundant with the structured data in song_lines. The rationale is threefold:
- The lyrics fields serve as a searchable corpus for the LIKE queries in the search engine.
- They provide a backup if the structured line data needs regeneration.
- They enable quick display of full lyrics in contexts where structured rendering is not needed (e.g., a print-friendly view).

4.15.4 Field: chords

Data Type: TEXT, NULLABLE
Purpose: Stores the raw chord data in its original [C] bracket format exactly as it was imported.

This field is an audit trail. It preserves the original input data before it was decomposed into the relational structure. If a chord parsing error is discovered months after import, the administrator can re-parse this raw text to regenerate the line_chords records without needing to locate the original source file.

4.15.5 Field: original_key

Data Type: VARCHAR(10), NULLABLE
Purpose: Stores the musical key in which the song was originally written (e.g., "G", "D", "Bb").

This information serves two functions:
- Display: The song view page shows the original key as a reference point.
- Transposition Reference: When the user transposes by +2 and the original key is G, the UI displays "Current Key: A (Original: G)" so the musician knows exactly how far they have shifted from the standard arrangement.

4.16 Deep Dive: Line Chords Table (The Architectural Cornerstone)

The line_chords table deserves special attention because it embodies the core innovation of the entire system. Every other design decision in the application exists to support the data model represented by this table.

4.16.1 The char_index Field: Mathematical Foundation

The char_index is a zero-based integer that represents a position in the line_text string of the corresponding song_line record.

Zero-Based Indexing Explained:
In the string "Amazing grace", the characters are indexed as:
   A=0, m=1, a=2, z=3, i=4, n=5, g=6, (space)=7, g=8, r=9, a=10, c=11, e=12

If a chord has char_index = 8, it means: "Display this chord above the character at position 8" which is the "g" in "grace."

Why Zero-Based: Both Java's String.charAt(index) and JavaScript's string[index] use zero-based indexing. By storing the index in the same convention, the application avoids off-by-one errors during both server-side processing and client-side rendering.

4.16.2 How char_index Survives Font Changes

Consider the traditional space-based approach:
   G        D
   Amazing  grace

If the font changes from monospaced (Courier) to proportional (Arial), the width of 8 space characters no longer equals the width of "Amazing " (8 letters). The "D" chord drifts left or right, landing over the wrong word.

With char_index = 8, the rendering engine does NOT use spaces for positioning. Instead:
1. It reads the lyric string character by character.
2. At position 8, it inserts a chord HTML element.
3. The chord element is physically embedded WITHIN the text flow, not floating above it.
4. Therefore, the chord ALWAYS appears exactly where position 8 is rendered, regardless of font.

4.16.3 How char_index Survives Screen Resizing

When the browser resizes and text wraps:
- Traditional approach: The chord line stays on one visual line while the lyrics wrap to two lines. Alignment destroyed.
- char_index approach: The Virtual Wrapping Algorithm splits the string at, say, position 20. Chords with char_index < 20 go to sub-line 1. Chords with char_index >= 20 go to sub-line 2 with adjusted positions (char_index - 20). Alignment preserved mathematically.

4.16.4 Real Data Example

Song: "Prabhu Tera Dhanyawad" (Hindi worship song)
Line text (Roman): "Prabhu tera dhanyawad karta hoon"
Line text (Devanagari): "प्रभु तेरा धन्यवाद करता हूँ"

Line chords:
| chord | char_index | Appears above (Roman) | Appears above (Devanagari) |
| D | 0 | "P" in Prabhu | "प्र" in प्रभु |
| G | 7 | "t" in tera | "ते" in तेरा |
| A | 12 | "d" in dhanyawad | "ध" in धन्यवाद |
| D | 22 | "k" in karta | "क" in करता |

The SAME char_index values work for both scripts because the chord positions were calculated from the Roman transliteration, and the Devanagari text was aligned to match the same musical timing points.

5.13 Chord Reconstruction: From Database to Screen

This section traces the exact transformation from raw database rows to the final visual display, using a concrete example.

5.13.1 Starting Point: Database Records

Given the following records:

song_lines record:
   id: 501, section_id: 201, line_text: "Amazing grace how sweet the sound", line_order: 1

line_chords records:
   id: 1001, line_id: 501, chord: "G", char_index: 0
   id: 1002, line_id: 501, chord: "C", char_index: 14
   id: 1003, line_id: 501, chord: "G", char_index: 24

5.13.2 Step 1: Java DAO Reads the Data

The SongDAO maps these records into Java objects:

   StructuredLine line = new StructuredLine();
   line.setLyrics("Amazing grace how sweet the sound");
   line.addChord(new ChordOccurrence("G", 0));
   line.addChord(new ChordOccurrence("C", 14));
   line.addChord(new ChordOccurrence("G", 24));

5.13.3 Step 2: JSP Generates Interleaved HTML

The JSP rendering function walks through the lyric string, splitting it at each chord position:

   Segment 1: Characters 0 to 13 = "Amazing grace " (preceded by chord G)
   Segment 2: Characters 14 to 23 = "how sweet " (preceded by chord C)
   Segment 3: Characters 24 to end = "the sound" (preceded by chord G)

Generated HTML:
   <div class="song-line">
     <span class="chord-wrap">
       <span class="chord-symbol">G</span>
     </span>Amazing grace
     <span class="chord-wrap">
       <span class="chord-symbol">C</span>
     </span>how sweet
     <span class="chord-wrap">
       <span class="chord-symbol">G</span>
     </span>the sound
   </div>

5.13.4 Step 3: CSS Positions Chords Above Lyrics

The CSS for chord-wrap uses position: relative, and chord-symbol uses position: absolute with bottom: 100%. This causes the chord text to float directly above the lyric text it precedes.

Visual result:
   G              C          G
   Amazing grace  how sweet  the sound

5.13.5 Step 4: JavaScript Virtual Wrapping (If Needed)

On a mobile screen (320px), the line "Amazing grace how sweet the sound" may not fit. The JavaScript engine:

1. Measures the container: 320px
2. Measures average character width: ~8px
3. Calculates characters per line: 320 / 8 = 40 characters (fits on this screen)
4. Since 33 characters < 40, NO wrapping needed on this particular line.

For a longer line like "I will sing of your love forever and ever amen hallelujah" (58 characters):
1. Characters per line: 40
2. Find last space before position 40: position 38 (after "amen")
3. Split into: "I will sing of your love forever and ever amen " and "hallelujah"
4. Redistribute chords based on which sub-line they belong to.

5.14 Comprehensive Edge Cases

5.14.1 Long Lines with Many Chords

Scenario: A line with 80 characters and 10 chord changes, viewed on a 320px phone.
Challenge: The line may split into 3 sub-lines, each needing its own subset of chords with recalculated positions.

How the system handles it:
The Virtual Wrapping Algorithm runs iteratively. After the first split at position 35, chords at positions 0-34 go to sub-line 1, and chords at positions 35-79 are carried forward. The algorithm then checks if sub-line 2 (positions 35-79, containing 45 characters) also needs splitting. It splits again at position 70. The result is three sub-lines with correctly distributed chords.

5.14.2 Two Chords at the Same Character Position

Scenario: The input "[G][Am]Lord, I need you" produces two chords at char_index 0.
Challenge: Both chords must be displayed above the word "Lord" without overlapping.

How the system handles it:
The CSS uses display: inline-flex on the chord-wrap elements. When two chord-symbol spans appear consecutively before the same lyric text, flexbox arranges them horizontally with a small gap (margin-right: 4px). The visual result is:
   G Am
   Lord, I need you

5.14.3 Lines with Only Chords (No Lyrics)

Scenario: An instrumental intro marked as "[G] - [C] - [D] - [G]"
After parsing: lyrics = " -  -  - ", chords = G@0, C@4, D@8, G@12
Challenge: The line has meaningful chord content but only dashes and spaces as "lyrics."

How the system handles it:
The rendering engine detects that the line_text consists only of whitespace and delimiter characters. It renders the line with a special "instrumental" CSS class that increases the chord font size and adds a subtle background to indicate that this line is for instruments only, not for singing.

5.14.4 Unicode Encoding Issues

Scenario: A Hindi song imported from a Windows system using ANSI encoding instead of UTF-8.
Challenge: Devanagari characters are garbled (displayed as "à¤†à¤°à¤¤à¥€" instead of "आरती").

How the system prevents this:
- The MySQL database is configured with CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci.
- The JDBC connection URL includes useUnicode=true&characterEncoding=UTF-8.
- The application deploys a CharacterEncodingFilter that sets request.setCharacterEncoding("UTF-8") and response.setCharacterEncoding("UTF-8") on every request.
- The JSP pages include the directive: <%@ page contentType="text/html; charset=UTF-8" %>

This four-layer encoding enforcement ensures that Unicode text is preserved from database to browser without corruption at any stage.

5.14.5 Chord Index Exceeding Line Length

Scenario: Due to a data import error, a chord has char_index = 50 but the line_text is only 30 characters long.
Challenge: Attempting to position a chord at index 50 on a 30-character string would cause an IndexOutOfBoundsException.

How the system handles it:
The rendering function validates each char_index before processing:
   if (occ.getPosition() >= lyrics.length()) {
       // Clamp to end of line
       occ.setPosition(lyrics.length());
   }
This safety check ensures that malformed data does not crash the rendering engine. The chord is displayed at the end of the line with a visual warning indicator for the administrator.

5.15 Complete User Flow: End-to-End Data Journey

This section traces every step of data as it moves from the user's action to the final pixels on screen.

5.15.1 Flow 1: Searching for a Song

User Action: Types "Aarati" into the search bar.

Step 1 - Client (JavaScript):
The keyup event fires. A 300ms debounce timer starts. After 300ms of no further typing, the script sends:
   fetch('/search?q=Aarati')

Step 2 - Network:
The HTTP GET request travels to the Tomcat server.

Step 3 - Server (SearchServlet):
The Servlet extracts the query parameter: String query = request.getParameter("q"). It calls searchService.search("aarati") after normalizing to lowercase.

Step 4 - Server (SearchService):
The service calls songDAO.searchSongs("aarati"). The DAO executes:
   SELECT id, song_number, title, artist, language FROM songs
   WHERE is_active = TRUE
   AND (LOWER(title) LIKE '%aarati%'
        OR LOWER(lyrics_roman) LIKE '%aarati%'
        OR LOWER(artist) LIKE '%aarati%')

Step 5 - Database (MySQL):
MySQL scans the indexed columns, finds 3 matching songs. Returns result set.

Step 6 - Server (SearchService):
The service calculates scores:
- Song "Aarati Ho Aarati" (title match): score 70
- Song "Prabhu Ki Aarati" (lyrics match): score 30
- Song "Aarati Karo" (title match): score 70
Sorts by score descending, then alphabetically for ties.

Step 7 - Server (SearchServlet):
Converts the sorted list to JSON:
   [{"id":45,"title":"Aarati Ho Aarati","score":70},
    {"id":78,"title":"Aarati Karo","score":70},
    {"id":112,"title":"Prabhu Ki Aarati","score":30}]

Step 8 - Network:
JSON response sent to browser.

Step 9 - Client (JavaScript):
Parses JSON. Creates HTML cards for each result. Inserts into the search results container. User sees 3 results within 200ms of finishing typing.


5.15.2 Flow 2: Viewing and Transposing a Song

User Action: Clicks "Aarati Ho Aarati" from search results.

Step 1 - Client: Browser navigates to /song?id=45.

Step 2 - Server (SongViewServlet):
Extracts id = 45. Calls songService.getFullSong(45).

Step 3 - Server (SongService -> SongDAO):
Executes the multi-JOIN query. Returns Song object with:
- 3 sections (Verse 1, Chorus, Verse 2)
- 8 lines total
- 22 chord occurrences

Step 4 - Server (SongService):
Checks session for transpose offset. Session has transpose = 0 (no transposition). Skips transposition step.

Step 5 - Server (JSP Rendering):
songView.jsp generates HTML: 3 section divs, each containing line divs with interleaved chord spans. Both data-original and data-roman attributes are set on each line.

Step 6 - Client (Browser):
HTML renders. JavaScript Virtual Wrapping runs. On a 390px phone, 2 of the 8 lines require splitting. Algorithm completes in 8ms.

Step 7 - User sees the song with perfectly aligned chords.

User Action: Taps "Transpose +1" button.

Step 8 - Client: JavaScript sends AJAX request: /song?id=45&transpose=1

Step 9 - Server: Session transpose value updated from 0 to 1. Song re-fetched. ChordTransposer shifts every chord by +1 semitone. C becomes C#, F becomes F#, G becomes G#.

Step 10 - Server: Returns updated HTML (or JSON with new chords).

Step 11 - Client: Re-renders chord symbols. Virtual Wrapping re-executes (chord widths may have changed due to # symbols). Total time: 30ms.

Step 12 - User sees transposed song. Display shows "Current Key: C# (Original: C)".

CHAPTER 6: TESTING AND VALIDATION

6.1 Testing Strategy

The Worship Chord and Lyrics Management System is a highly dynamic application where musical data must be precisely aligned and mathematically manipulated in real-time. A generic testing approach was insufficient for validating the system's correctness. The testing strategy was designed around the three core technical innovations:

Unit Testing: Validates the mathematical correctness of individual algorithms (ChordTransposer, ChordParser) in isolation, without requiring a running server or database connection.

Integration Testing: Validates the complete data flow from database to rendered output, ensuring that structured relational data (Songs -> Sections -> Lines -> Chords) is correctly fetched, assembled, and delivered.

Rendering Testing: Validates the Visual Wrapping Algorithm across multiple viewport sizes and multi-script character sets, ensuring that chords maintain perfect alignment under all display conditions.

Performance Testing: Measures response times for search queries, song loads, and transposition operations to ensure they meet the sub-100ms threshold required for live performance environments.

Security Testing: Validates that the system is immune to common web application vulnerabilities including SQL injection and cross-site scripting.

6.5 Edge Case Testing

Edge cases represent unusual or extreme inputs that could cause unexpected behavior. Thorough edge case testing is essential for production readiness.

6.5.1 Empty Song Lines

Scenario: Songs with instrumental breaks that have chord symbols but no lyrics.
Input: "[Intro: G - C - D]" followed by empty lyric lines.
Expected: Chords displayed on a standalone line without lyrics below.
Result: PASS. The rendering engine creates a transparent placeholder element to hold the chords.

6.5.2 Multiple Chords at Same Position

Scenario: A rapid chord change occurs at the beginning of a word, such as [G][C]Lord.
Expected: Both chords should be displayed at position 0 without overlapping the lyrics.
Result: PASS. CSS flexbox rules stack the chords horizontally (G C) above the word "Lord" with appropriate spacing.

6.5.3 Extreme Transposition (24 Steps)

Scenario: User rapidly clicks the "+1" transpose button 24 times in succession.
Expected: The chord should return to its original value after 12 clicks (full circle) and then cycle again. No errors should occur.
Result: PASS. The modulo arithmetic in ChordTransposer handled the wrapping correctly. After 12 clicks: C -> C. After 24 clicks: C -> C. No IndexOutOfBoundsException or unexpected behavior.

6.5.4 Very Long Song (100+ Lines)

Scenario: A song with over 100 lines and 300+ chord occurrences.
Expected: The rendering engine should complete within the 50ms performance threshold.
Result: PASS. Rendering completed in 28ms. DOM manipulation was batched using document fragments.

6.5.5 SQL Injection Attempt

Scenario: A user enters " OR 1=1;-- " as a search query.
Expected: The query should be treated as a literal string, not as SQL code.
Result: PASS. The PreparedStatement parameterization neutralized the injection. The search returned zero results (no song matches the literal string " OR 1=1;--").
